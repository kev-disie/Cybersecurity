# SECTION A – QUESTION 1 (30 MARKS)

## a) Four Types of Errors During Compilation (4 Marks)

1. **Lexical Error** – Invalid character/token. Example: `int 9x = 5;` (identifier starts with digit)
2. **Syntax Error** – Violation of grammar rules. Example: `if (x > 5` (missing closing parenthesis)
3. **Semantic Error** – Grammatically correct but meaningless. Example: `int x = "hello";` (type mismatch)
4. **Logical Error** – Program runs but gives wrong output. Example: using `+` instead of `*` in a formula

---

## b) Three Rules of Code Optimization (3 Marks)

1. **Constant Folding** – Evaluate constant expressions at compile time (e.g., `x = 3 * 4` → `x = 12`)
2. **Dead Code Elimination** – Remove code that never executes
3. **Common Subexpression Elimination** – Avoid recomputing the same expression twice

---

## c) Three Modes of Intermediate Representation (6 Marks)

1. **Three-Address Code (TAC)** – Each instruction has at most one operator and three addresses.
   Example: `t1 = a + b; t2 = t1 * c`

2. **Syntax Tree / Abstract Syntax Tree (AST)** – Tree representation where internal nodes are operators and leaves are operands.
   Example: `a + b * c` forms a tree with `+` at root, `a` as left child, `*` as right with `b`, `c`

3. **Postfix Notation (Reverse Polish Notation)** – Operators follow operands.
   Example: `a + b * c` → `a b c * +`

---

## d) Parse Tree vs Syntax Tree for `a a + a *` using grammar (4 Marks)

**Grammar:**

## e) Three Differences: Compiler vs Interpreter (6 Marks)

| | **Compiler** | **Interpreter** |
|---|---|---|
| 1 | Translates entire program before execution | Translates and executes line by line |
| 2 | Produces an executable object file | No object file is produced |
| 3 | Faster execution after compilation | Slower — re-translates every run |

## f) Copy Propagation on the Code (3 Marks)

**Original:**
```
b := z + y
a := b
x := 2 * a
```

**After Copy Propagation** (replace `a` with `b`, then `b` with `z+y`):
```
b := z + y
x := 2 * b       ← a replaced by b
x := 2 * (z + y) ← b replaced by z+y (further propagation)
```
`a` and `b` become unnecessary and can be eliminated.

---

## g) Definitions (4 Marks)

**i. Translator** – A program that converts source code from one language to another (e.g., compiler, assembler, interpreter).

**ii. Lexeme** – The actual character sequence in source code that matches a token pattern. Example: `total` is a lexeme of token type `IDENTIFIER`.

**iii. Pattern** – A rule (often a regular expression) describing the form a lexeme can take. Example: `[a-zA-Z][a-zA-Z0-9]*` is the pattern for identifiers.

**iv. Tokenization** – The process of scanning source code and breaking it into a sequence of tokens (meaningful units like keywords, identifiers, operators).

## b) Language Processing System (10 Marks)

```
Source Program
      ↓
[Preprocessor]  ← handles macros, includes
      ↓
Modified Source
      ↓
[Compiler]      ← lexical, syntax, semantic analysis → code generation
      ↓
Target Assembly Code
      ↓
[Assembler]     ← converts assembly to machine code
      ↓
Relocatable Object Code
      ↓
[Linker/Loader] ← links libraries, loads into memory
      ↓
Executable Machine Code
      ↓
[Output / Running Program]
```

Each stage transforms the program closer to machine-executable form.

## c) Code Optimization on `add_ten` (4 Marks)

**Original:**
```c
int add_ten(int x) {
    int y, z;
    y = 10;
    z = x + y;
    return z;
}
```

**Optimized** (constant propagation + copy propagation):
```c
int add_ten(int x) {
    return x + 10;  // y=10 folded in, z eliminated
}
```

Optimizations applied: **Constant Propagation** (y→10) and **Dead Variable Elimination** (y, z removed).

# QUESTION 4 (20 MARKS)

## a) Three Components of a Type Checker (6 Marks)

1. **Type Inferencer** – Determines the type of each expression based on operands and operators (e.g., `int + int = int`)
2. **Type Validator** – Checks that types used in operations are compatible (e.g., no `int + string`)
3. **Type Converter / Coercion Handler** – Handles implicit type conversions where allowed (e.g., `int` to `float`)

## c) Two Error Recovery Schemes (4 Marks)

1. **Panic Mode Recovery** – The parser discards input tokens one by one until a synchronizing token (e.g., `;`, `}`) is found, then resumes parsing. Simple but may skip large portions.

2. **Phrase Level Recovery** – The parser performs local correction on the remaining input (e.g., inserting a missing semicolon or replacing a token) to allow parsing to continue.
